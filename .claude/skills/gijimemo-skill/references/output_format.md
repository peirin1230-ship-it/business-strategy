# 議事メモ 出力フォーマット

以下のフォーマットに従って議事メモを出力する。各セクションの目的と記載ルールを厳守すること。

---

## 完全フォーマット

```markdown
# 議事メモ

## 会議情報
| 項目 | 内容 |
|------|------|
| 日時 | YYYY/MM/DD HH:MM〜HH:MM |
| 会議名 | （会議名） |
| 参加者 | （名前・役職。意思決定者には ★ を付与） |
| 記録者 | （記録者名） |
| 配布先 | （このメモを共有すべき人） |

---

## 大論点（BIG QUESTION）

> **この会議で答えるべき問い：**
> 「〜すべきか？」「〜はどうあるべきか？」の形で記述

---

## 論点別ディスカッションサマリー

### 論点1: （論点を「問い」の形で記述）

**事前仮説：**
（会議前に想定していた仮説。入力テキストから推定する場合は「（推定）」と注記）

**議論の要旨：**
（主要な発言・ファクトを構造的に整理。発言者名を明記）

**空気感：**
（この論点に対する参加者の温度感。例：「A部長は明確に賛成（前のめり）」「B課長はしぶしぶ賛成（懸念を残しつつ）」「C氏は態度保留（情報不足を指摘）」）

**仮説の進化：**
（事前仮説がこの議論を経てどう変わったか）
- Before: （議論前の認識）
- After: （議論後の認識）
- 変化のポイント: （何がきっかけで仮説が変わったか）

**ステータス：** 決着 / 未決 / 要追加検討

---

（同構造で繰り返し）

**TASKではなく、論点（「〜すべきか？」の形）として書くこと。**

1. （次に解くべき論点1）

   - 仮説: 現時点での仮説（あれば）

   - 背景:
   - 仮説:

## 空気感メモ

この会議全体のトーン・温度感を一言〜数行で記述。

（例：「全体的に前向きなトーン。ただしコスト面の懸念が根強く、CFOのX氏は終始慎重姿勢。次回のデータ次第で空気が変わる可能性あり」）
```

## 記載ルール

### 論点の書き方
- 必ず「問い」の形にする（「〜すべきか？」「〜はどうか？」）
- 悪い例：「予算について」「スケジュール確認」（これはテーマであって論点ではない）
- 良い例：「追加予算2000万円を承認すべきか？」「Q3までのローンチは実現可能か？」

### 空気感の書き方
- 単に「賛成/反対」ではなく温度感を表現する
- 表現例：「前のめり」「慎重ながらも賛成」「態度保留」「明確に反対（感情的）」「ロジカルに反対（代案あり）」「沈黙（≒消極的反対の可能性）」
- 入力テキストから読み取れない場合は「（※入力情報から空気感を推定できず）」と正直に書く

### 仮説の進化の書き方
- Before/Afterの形式で変化を明示する
- 「変化なし」も重要な情報。その場合は「事前仮説が補強された」と記載
- 入力テキストに事前仮説の情報がない場合は、議論の流れから推定し「（議論の流れから推定）」と注記

### ネクスト論点 vs ネクストステップ
- **ネクスト論点**：「〜すべきか？」という問い。思考の方向性を示す
- **ネクストステップ**：具体的なTASK。手を動かす内容
- 必ずネクスト論点を先に書き、それに紐づくTASKを後に書く
- TASKだけ書くのは「TASKバカ」（高松氏）

### 入力情報が不足している場合
- 推定で補える部分は「（推定）」「（議論の流れから推定）」と注記して記載
- 推定すら困難な部分は「（※情報不足のため記載不可。確認推奨）」と明示
- 空欄にするよりも、推定＋注記のほうがはるかに価値が高い
